## 3. The B3 Test — What It Found and How We Got Here

B3 refers to Strategy C, the third run of the Lisbon methodology test (March 2026). Understanding B3 is essential to understanding why the proposed definition change is the right move — and not a regression.

### The three Lisbon strategies

| Test | Approach | Both-pool found | Verdict |
|------|----------|----------------|---------|
| B1 (Strategy A) | Structured baseline, 30 queries | 23 BP | Inflated — all from Time Out EN + Time Out PT (same publisher) |
| B2 (Strategy B) | Writer-first emphasis, 36 queries | 8 BP | Genuine independent sources but lower restaurant count |
| B3 (Strategy C) | Phase 0 direct fetches + COI check + publisher concentration check | 0 BP | Honest structural finding — independent EN/PT critics cover different restaurant segments |

### What B3 found

Strategy C found that genuinely independent English-language and Portuguese-language critics in Lisbon do not recommend the same restaurants. This is not a pipeline failure — it is an accurate reflection of how different editorial communities engage with Lisbon's food scene: EN writers (expat/tourist-facing) cover different restaurants from PT critics (local audience). The zero both-pool count is honest, not a gap.

### What B3 triggered

B3 confirmed that Strategy A's 23 both-pool entries were entirely false positives: Time Out EN and Time Out PT are the same publisher (Time Out Group) writing about the same restaurants in two languages. This is one organisation's recommendations, not independent consensus. The both-pool definition was therefore tightened:

**Previous definition:** Restaurant recommended by sources from two different language pools.

**Post-B3 definition (current):** Restaurant recommended by two editorially independent organisations from different publishers, writing in different languages. Same-publisher cross-language editions do not qualify.

### Why the proposed definition is the next correct step

The current (post-B3) definition solved the false positive problem. But it created a new gap: two genuinely independent Italian publishers (e.g. Gambero Rosso and CiboToday) recommending the same Roman trattoria cannot create both-pool, because they write in the same language. That is real editorial consensus being excluded by an arbitrary language requirement.

The writer discovery test across all 41 cities confirms that:

1. Every non-BP restaurant in the fleet has source pairs from publishers that are already genuinely independent.
2. The only barrier to both-pool status for these restaurants is the language requirement.
3. No city has false pairs that would be incorrectly unlocked (publisher independence check prevents that).
4. B3's core fix is preserved: Time Out EN + Time Out PT still disqualified (same publisher).

The proposed definition — publisher independence only, language irrelevant — is therefore a refinement of B3's finding, not a reversal of it. Language was never the signal. Publisher independence was always the signal. B3 identified it. The proposed change makes it the sole criterion.

## 5. Structural Patterns Confirmed Fleet-Wide

### Pattern A: Turin/Porto pattern (source-restaurant coverage divergence)

Cities with 0 BP despite having sources in multiple languages — because the cross-language sources cover entirely different restaurants rather than overlapping. The language requirement cannot be met regardless of how many sources exist.

Affected cities: **Turin** (IT sources cover different restaurants than EN sources), **Porto** (Visão/PT covers different restaurants than EN sources), **Seville** (structural gap, no overlap). Definition change fixes Turin and Porto. Seville requires investigation.

### Pattern B: Same-language different-publisher pairs (the unlock)

Cities where all non-BP restaurants have two-source pairs, but both sources write in the same language. These are genuine independent recommendations blocked only by the language requirement.

Affects: All non-BP 2-source restaurants in Spain, Italy, Portugal, Morocco except those in the Turin/Porto pattern above.

### Pattern C: Single-source structural ceiling

Cities where most or all restaurants have only one source. No definition change helps. Requires editorial rebuild.

Affected cities: **Palermo** (18R, all single-source), **Chefchaouen** (2R), **Vigo** (all URLs null, source quality uncertain).

## 6. UI Modifications Required

**Three changes must happen simultaneously** to avoid inconsistency between old and new cities:

1. **Pipeline template** — remove language requirement from both_pool rule. Different publisher only.
2. **postrun.js Step 1.5** — change logic: set "both" if 2+ distinct publication values (publisher diversity, not language diversity).
3. **Recomputation script** — write once, run across all 41 city JSONs to recompute both_pool from new rule.

### Viewer copy changes (grep targets in index.html)

| Element | Current | Required change | Effort |
|---------|---------|-----------------|--------|
| Card badge | "Both" | Keep or rename "Consensus" — stop implying language | Low |
| Filter dropdown | "Both languages" | "Consensus picks" | Low |
| Sources panel explanation | Cross-language framing | "Recommended by N editorially independent sources from different publishers" | Medium |
| Tooltip / hover text | Any "cross-language" reference | "cross-publisher" framing | Low |
| Landing page / hero copy | Language diversity as quality signal | Publisher independence / editorial independence | Medium |
| `language_pool` field | Values: "en", "fr", "pt", "both" | Field name stays (defer to Next.js migration). Value "both" now means cross-publisher. Add code comment only. | Low |
| postrun.js Step 1.5 | Derives from language diversity | Derives from publisher diversity | Medium |
| Pipeline template | Different publisher + different language | Different publisher only | Low |
| Recomputation script | Does not exist | Write once, run across all 41 cities | High |

### What to expect from the UI exercise

**Most viewer copy changes = 1–3 line edits in index.html.** The language framing appears in a small number of places — a grep for "cross-language", "both language", "languages" will find them all. These are the lowest effort changes in the whole exercise.

**The recomputation script = high effort but one-time.** Once written, it reads each city JSON, applies the new both_pool rule (publisher independence only), and writes updated both_pool values. Run time is minutes across all 41 cities. The script itself is the only meaningful engineering task.

**The three simultaneous changes are the coordination constraint.** If done piecemeal — e.g. updating the viewer before updating the pipeline template — new cities added during the transition period will be computed under the old rule while existing cities display under the new rule. All three changes should land in the same deployment.

**Expected net viewer output after the exercise:**
- Fleet BP count: 119 → ~248 (19.2%)
- City-level cards: most cities gain 2–11 BP badges
- Sources panel: copy update makes the "consensus" framing clearer and more accurate
- No visual design changes required — the badge, filter, and panel structure stay the same; only the copy and the underlying logic change
- Palermo, Chefchaouen, Toronto: unchanged (structural gaps, no fix from definition change)
