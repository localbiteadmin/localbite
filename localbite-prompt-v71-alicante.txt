STEP 1 — Write search plan

Before running any searches, write a plan to OUTPUT_SEARCH_PLAN.

MANDATORY SELF-CONFIRMATION — write this as the first line
of OUTPUT_SEARCH_PLAN before anything else:
"MANDATORY REQUIREMENTS CONFIRMED: year range distribution
across [YEAR_RANGE], cross-validation Phase 1B after main
searches, audit file required, 10 results per query (hard limit)."
If you cannot confirm all four, stop and report which
requirement cannot be met before proceeding.

The search plan has two parts:

PART A — NEIGHBOURHOOD COVERAGE (non-negotiable)

Structure queries using the tier variables from Part 1:

TIER_1_NEIGHBOURHOODS (central / tourist areas):
For each named area: 1 query in PRIMARY_LANGUAGE +
1 query in SECONDARY_LANGUAGE = 2 queries per area.

TIER_2_NEIGHBOURHOODS (residential areas):
For each named area: 1 query in PRIMARY_LANGUAGE +
1 query in SECONDARY_LANGUAGE = 2 queries per area.

TIER_3_NEIGHBOURHOODS (emerging areas):
For each named area: 1 query in PRIMARY_LANGUAGE only.
If TIER_3_NEIGHBOURHOODS is blank: skip Tier 3 entirely.

Every neighbourhood must be named explicitly in at least
one query. Do not substitute or add areas not in the tiers.

PART B — CROSS-CUTTING ANGLES (run after all neighbourhood queries)
These angles cover what neighbourhood queries miss:

  Cross-1 — City-wide in SECONDARY_LANGUAGE: 2 queries
    Vary register — one editorial guide style, one personal
    recommendation style. Do not reuse neighbourhood phrasing.

  Cross-2 — City-wide in PRIMARY_LANGUAGE: 3 queries
    Use 3 meaningfully different registers: formal food
    criticism, casual neighbourhood recommendation, food
    journalism / press guide.

  Cross-3 — Generic writer-first: 2 queries
    Open discovery queries. Do not name specific writers.
    1 in each language.

  Cross-4 — Named awards: 3 queries
    Use award names from NAMED_AWARDS exactly. One query per
    award. Target articles about results and winners — not
    the award body itself. If NAMED_AWARDS is blank, run
    generic award queries: 1 EN + 1 local language + 1 Michelin.

  Cross-5 — Recent openings: 2 queries
    1 in each language. Target new openings specifically.

  Cross-6 — Vocabulary-specific local language: 2 queries
    Use vocabulary from LOCAL_FOOD_VOCABULARY. Target the
    casual neighbourhood register city-wide queries miss.
    If LOCAL_FOOD_VOCABULARY is blank: skip this angle entirely.

  Cross-7 — Cuisine and community: 2 queries
    Target food communities whose writing is not captured
    by generic city-wide queries. Use specific cuisine types
    or cultural communities relevant to this city.
    If not applicable: skip this angle.

  Cross-8 — Publication-category local language: 2 queries
    Target categories of local publication by descriptor only.
    Do not name specific publications.
    1 independent food magazine / revista gastronómica
    1 newspaper gastronomy supplement

- Article is specifically about this city's restaurants
- Page returns 2,000+ chars confirming genuine article content

If under 2,000 chars or no date visible: reject without using
a full fetch slot.

Record the exact Jina fetch URL as article_url for every
source that passes verification. This is the specific article
URL — not the publication homepage. It is required for the
source record and will be used as the viewer source link.

SOURCE QUALITY RULES:
✓ Full name author in article text, date within YEAR_RANGE,
  restaurant guide for this city, editorial independence,
  no obvious advertorial
✓ Personal single-author site — About page byline of full
  name sufficient (apply single-author rule — log confirmation)
✓ Independent blogger with evidence of local residency or
  regular city coverage
✓ Freelance food writer with documented city coverage
✓ Local press journalist writing a food guide for this city
✗ No named author anywhere — reject regardless of publication
✗ Initials-only byline (e.g. "Irene S.", "J. Smith") — REJECT
✗ Date outside YEAR_RANGE (verify from fetched text, not
  updated label or URL)
✗ Booking sites, hotel sites, sponsored content
✗ Covers whole country or region, not this city specifically
✗ KNOWN_PROBLEMATIC_URLS
✗ Social media primary sources without fetchable articles
✗ Articles that read as promotional or lack specific critical detail

CONFLICT OF INTEREST CHECK (mandatory for every source):

FETCH PROTOCOL:

that source. Record it precisely — this is what the viewer
will link to when a user clicks "Source ↗".

TWO-TIER RETRY RULE:
If any fetch returns under 3,000 characters:

  RETRY 1 — Transient failure check:
  1. Wait 30 seconds
  2. Retry the same URL via Jina
  3. If now 3,000+ chars: proceed normally.
     Log: [retry-1-success] [publication] — recovered after 30s

  RETRY 2 — Extended wait (if Retry 1 still under 3,000 chars):
  1. Wait 60 seconds
  2. Retry the same URL via Jina one final time
  3. If now 3,000+ chars: proceed normally.
     Log: [retry-2-success] [publication] — recovered after 60s

  FINAL FAILURE (if still under 3,000 chars after both retries):
  - Mark as fetch_status: "failed"
  - Try Wayback Machine fallback:
    https://r.jina.ai/https://web.archive.org/web/2026/[URL]
    Log: [wayback-retry] [publication] — [outcome]
  - If Wayback also fails: log to OUTPUT_FAILED_SOURCES and skip
  - writer_profile must still be written from search snippet

END-OF-PHASE FAILED SOURCES REPORT:
After all sources processed, output:
FAILED SOURCES SUMMARY:
- Sources recovered by Retry 1 (30s): [list]
- Sources recovered by Retry 2 (60s): [list]
- Sources recovered by Wayback: [list]
- Sources failed after all retries: [list with char counts]
- Total restaurants lost to permanent failures: [estimate]

For each source report:
- Publication and writer
- Characters returned (attempt 1 / retry 1 / retry 2 / wayback)
- Fetch quality: good (3,000+) / partial (1,000–3,000) /
  failed (under 1,000 after all retries)
- Retries attempted: 0 / 1 / 2
- Wayback attempted: yes/no
- Outcome: success / recovered / failed
- Restaurants extracted (count, or 0 if failed)

STEP 2 — Assign confidence tier

BOTH-POOL DEFINITION (strict):
A restaurant qualifies as both-pool ONLY if it appears in:
- At least one source in language pool A, AND
- At least one source in language pool B, AND
- The two sources are from DIFFERENT publishers with no shared

- 2 or more independent sources, OR
- Both-pool (using strict definition above)

- Single primary or independent source, good fetch quality,

- Secondary single source, good fetch quality, quote ≥15 words
  with ≥2 specific details

- Failed fetch, single source

- Failed traceability check
