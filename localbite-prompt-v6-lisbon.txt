PHASE 1 — SOURCE SELECTION
════════════════════════════════════════

STEP 1 — Write search plan

Before running any searches, write a plan to OUTPUT_SEARCH_PLAN.

The search plan has two parts:

PART A — NEIGHBOURHOOD COVERAGE (non-negotiable)
Run one search query per neighbourhood or neighbourhood group
listed in GEOGRAPHIC_BOUNDARY_INCLUDE. Adjacent neighbourhoods
with overlapping food writing scenes may be grouped into one
query (e.g. "Corktown and Distillery District Toronto restaurants
food writer" covers both). Every neighbourhood must appear in at
least one query — do not skip any.

Run neighbourhood queries in all relevant SEARCH_LANGUAGES.
For a bilingual city: run each neighbourhood in both languages
(2 queries per neighbourhood or group). For a trilingual city:
run in all 3 languages (3 queries per neighbourhood or group).
For a single-language city: 1 query per neighbourhood or group.

PART B — CROSS-CUTTING ANGLES (run after all neighbourhood queries)
These angles cover what neighbourhood queries miss:

  Cross-1 — City-wide: 3 queries (vary language and year terms)
  Cross-2 — Writer-first: 3 queries (use any names from Phase 0;
    vary language; include one query per language pool)
  Cross-3 — Recent openings 2024–2026: 2 queries
  Cross-4 — Older coverage 2023–2024: 2 queries (explicitly
    search for "best restaurants [city] 2023" and "2024" — do
    not skip; established sources from these years are valid)
  Cross-5 — Awards and recognition: 2 queries (Michelin, local
    guides, annual best-of lists)
  Cross-6 — Forum and community: 2 queries
  Cross-7 — Cuisine diversity (for cities with significant
    non-mainstream food communities): 2 queries if applicable,
    0 if not

TOTAL QUERY COUNT:
  Small city (≤5 neighbourhoods, e.g. Fes, Chefchaouen):
    ~5–15 neighbourhood queries + 14 cross-cutting = ~20–30 total
  Medium city (5–15 neighbourhoods, e.g. Lisbon, Rabat, Valencia):
    ~15–30 neighbourhood queries + 14 cross-cutting = ~30–45 total
  Large city (15+ neighbourhoods, e.g. Toronto, Barcelona):
    ~20–40 neighbourhood queries + 14 cross-cutting = ~35–55 total

STOPPING RULE:
If the last 8 consecutive queries have all returned sources
already in the candidate pool or no qualifying NEW SOURCES at all
(a query surfacing known sources but no new ones counts as
yielding nothing new), stop searching and proceed to Step 3 —
even if not all neighbourhood queries or angles are complete.
Log: [stopping-rule] triggered after [N] queries — last 8 queries
yielded no new candidates.

The stopping rule takes precedence over neighbourhood completeness.
For well-documented cities (european_major), the source pool
typically saturates after 30–40 queries — the stopping rule will
fire before all neighbourhood combinations are exhausted, and this
is correct behaviour. For thin-source cities (moroccan_medina,
european_minor), the stopping rule may not fire until much later.

Soft ceiling: 55 queries. If approaching 55 without triggering
the stopping rule, complete the current angle and stop.
Do not exceed 55 under any circumstances — context window risk.

QUERY RULES:
✗ Each query must be meaningfully distinct from all previous
✗ Do not repeat any query already in OUTPUT_SEARCH_LOG
✓ Vary year terms across queries (some with years, some without)
✓ Vary language across queries using all SEARCH_LANGUAGES
✓ Neighbourhood queries should name the neighbourhood explicitly

Write the full plan before running any queries.

Run all planned queries in order: neighbourhood queries first

Do not skip Part A neighbourhood queries for any reason —

For each candidate found via search, run a quick verification
fetch via Jina to confirm:
- Publication date is 2023–2026 (check article text, not

- Named author present
- Article is specifically about this city's restaurants
- Page returns 2,000+ chars confirming genuine article content

If under 2,000 chars or no date visible: reject without using
a full fetch slot.

SOURCE QUALITY RULES:
✓ Named author, 2023–2026 date, restaurant guide for this city,
  editorial independence, no obvious advertorial framing
✓ Independent blogger with evidence of local residency or regular
  city coverage
✓ Freelance food writer with documented city coverage
✓ Local press journalist writing a food guide for this city
✗ No named author — reject regardless of publication credibility
✗ Pre-2023 date (verify from fetched text, not updated label)
✗ Booking sites, hotel sites, sponsored content
✗ Covers whole country or region not this city specifically
✗ KNOWN_PROBLEMATIC_URLS
✗ Social media primary sources without fetchable articles
✗ Articles that read as promotional or lack specific critical detail

WRITER PROFILE (mandatory for every accepted source):

PUBLISHER CONCENTRATION CHECK:
After selecting sources, check: does any single publisher
contribute 2+ articles (regardless of whether different writers
authored them) or 50%+ of either language pool?
IMPORTANT: A publication with two different writers is still one

If yes: log [publisher-concentration] flag, note the publisher and
article count, and attempt 3–4 supplementary searches targeting
independent sources from a different publisher before proceeding.
If no independent alternative found after supplementary searches,
document and proceed — this is a structural characteristic of
the city's food writing landscape, not a pipeline failure.

✓ At least 1 primary source from EACH local language pool listed

✓ No more than 2 secondary sources in total

✓ At least 2 legacy/institutional sources
✓ At least 1 independent/freelance source
✓ No more than 2 secondary sources in total

DIVERSITY GATE section of the final summary.

Select 6–12 sources. For each state: publication, writer,

Do not stop for approval. Proceed to Phase 2.

PHASE 2 — FETCH AND EXTRACT (with retry)
════════════════════════════════════════

FETCH PROTOCOL:
Fetch all sources via Jina (https://r.jina.ai/[URL]).
Send all Jina requests in parallel. Then process extractions
one source at a time in the order they return.

TWO-TIER RETRY RULE (new in v6):
If any fetch returns under 3,000 characters:

  RETRY 1 — Transient failure check:
  1. Wait 30 seconds
  2. Retry the same URL via Jina
  3. If now 3,000+ chars: proceed with extraction normally.
     Log: [retry-1-success] [publication] — recovered after 30s

  RETRY 2 — Extended wait (if Retry 1 still under 3,000 chars):
  1. Wait 60 seconds
  2. Retry the same URL via Jina one final time
  3. If now 3,000+ chars: proceed with extraction normally.
     Log: [retry-2-success] [publication] — recovered after 60s

  FINAL FAILURE (if still under 3,000 chars after both retries):
  - Mark as fetch_status: "failed"
  - Log to OUTPUT_FAILED_SOURCES:
      URL | chars attempt 1 | chars retry 1 | chars retry 2 |
      outcome: permanent failure
  - Do not extract restaurants from this source
  - Continue to next source
  - Source remains in sources array with fetch_status: "failed"
    and fetch_chars recording the final char count
  - writer_profile must still be written from search snippet
    knowledge — do not include fetch failure notes in this field

END-OF-PHASE FAILED SOURCES REPORT:
After all sources are processed, before proceeding to
traceability check, output:

FAILED SOURCES SUMMARY:
- Sources recovered by Retry 1 (30s): [list]
- Sources recovered by Retry 2 (60s): [list]
- Sources failed after both retries: [list with char counts]
- Total restaurants lost to permanent failures: [estimate]

If OUTPUT_FAILED_SOURCES is non-empty: the operator should
inspect these URLs after the pipeline completes. Permanent
failures are likely paywalls or scraper blocks. No further
automated action is taken — this is a known limitation.

For each source, follow this sequence:
1. Confirm fetch received and char count
2. If under 3,000 chars: apply two-tier retry rule above
3. If 3,000+ chars (initial or after retry): extract restaurants
4. Write to OUTPUT_WORKING (append)
5. Confirm write succeeded
6. Move to next source

For each source report:
- Publication and writer
- Characters returned (attempt 1 / retry 1 / retry 2)
- Fetch quality: good (3,000+) / partial (1,000–3,000) /
  failed (under 1,000 after all retries)
- Retries attempted: 0 / 1 / 2
- Outcome: success / recovered-retry-1 / recovered-retry-2 /

EXTRACTION RULES:
✓ Only include restaurants explicitly named in fetched text
✓ Extract: name, neighbourhood (canonical from
  GEOGRAPHIC_BOUNDARY_INCLUDE), price level, signature dish
  if mentioned, quote (writer's words, minimum 15 words with
  at least 1 specific detail), cuisine type, opening year if
  mentioned, article_date (from fetched text not metadata)
✗ Do not use training knowledge to add any detail
✗ Do not include if only evidence is a list title or header
✗ Secondary sources: maximum 30 restaurants

QUOTE QUALITY RULE:
Valid quote = at least 15 words AND at least one specific detail
(dish name, chef name, physical description, price, year, or
neighbourhood).
Under 15 words or no specific detail → low-confidence flag.
Secondary source entries → must have TWO specific details or
→ low-confidence.

ARTICLE DATE FIELD:
Record the date found in fetched article text as article_date.
If no date visible in fetched text, record as "undated".
Do not use CMS metadata, "updated" labels, or URL patterns.
If article_date is 2023 or 2024, set open_status_check: true
on every restaurant extracted from that source.

NAME NORMALISATION:
If the same restaurant appears under multiple spellings, merge
into one entry using the most complete name. Record all variants
in name_variants. Report all merges at end of phase.

NEIGHBOURHOOD STANDARDISATION:
Map to canonical forms from GEOGRAPHIC_BOUNDARY_INCLUDE.
Use null if neighbourhood cannot be determined from article text.
Do not invent neighbourhoods not in GEOGRAPHIC_BOUNDARY_INCLUDE.

LANGUAGE_POOL FIELD:
Set language_pool as a plain string — never a list or array.
Rules:
  "both"         → restaurant appears in sources from 2+ different
                   language pools (e.g. EN + ES, EN + CA, ES + CA).
                   Same-publisher cross-language editions do NOT qualify.
  "en"           → English-language sources only
  "es"           → Spanish-language sources only
  "ca"           → Catalan-language sources only
  "pt"           → Portuguese-language sources only
  "fr"           → French-language sources only
  "ar"           → Arabic-language sources only
  "multi-source" → SINGLE_LANGUAGE_CITY only: appears in 2+ independent
                   sources in the same language
Calculate this field in Phase 3 after all sources are merged.
Never write ["en", "es"] or any bracket/list form — always a string.

GEOGRAPHIC BOUNDARY:
✗ Auto-exclude anything in GEOGRAPHIC_BOUNDARY_EXCLUDE
✗ Flag any restaurant whose location description is ambiguous

RIAD RULE (apply only if RIAD_RULE: APPLY in Part 1):
Flag any hotel or riad restaurant with ⚠ riad.
Include only if: quote contains specific dish detail AND writer
explicitly recommends the restaurant for food, not atmosphere.
Remove if quote is atmosphere-only with no dish specificity.

TRACEABILITY CHECK (mandatory before proceeding to Phase 3):
After all sources are extracted, verify:
For every restaurant in OUTPUT_WORKING, can you identify the
specific source URL it came from and the sentence in the fetched
text where its name appeared?
If any restaurant cannot be traced to a specific fetched
sentence: remove it and log the removal in OUTPUT_AUDIT.
This check prevents fabrication. Do not skip it.

STEP 2 — Assign confidence tier

BOTH-POOL DEFINITION (v6 — strict):
A restaurant qualifies as both-pool ONLY if it appears in:
- At least one source in language pool A, AND
- At least one source in language pool B, AND
- The two sources are from DIFFERENT publishers with no shared
  ownership, brand relationship, or licensing arrangement.
Same-publisher cross-language editions (e.g. Time Out EN + Time
Out ES, Time Out EN + Time Out PT) do NOT qualify as both-pool
even if they have different editorial teams or different authors.

TIER A — Auto-approve:
- 2 or more independent sources, OR
- Both-pool (using strict definition above)

For single-language cities (SINGLE_LANGUAGE_CITY: YES):
- Tier A = at least one legacy/institutional source AND at least
  one independent/freelance source (record as "multi-source")
- Single-tier two-source entries are strong but not Tier A

TIER B — Review recommended:
- Single primary or independent source, good fetch quality,
  quote ≥15 words with ≥1 specific detail
- Secondary single source, good fetch quality, quote ≥15 words
  with ≥2 specific details

TIER C — Auto-reject:
- Failed fetch, single source (partial fetches that didn't
  meet retry threshold are treated as failed for this purpose)
- Quote under 15 words or no specific details
- Secondary single-source with fewer than 2 specific details
- Outside GEOGRAPHIC_BOUNDARY_INCLUDE
- Low-confidence flag from extraction
- Any entry from KNOWN_PROBLEMATIC_URLS
- Failed traceability check

SOURCE CONCENTRATION CAP:
If any single source contributes more than 30% of total
(Tier A + Tier B) entries as single-source Tier B:
Auto-reject lowest-confidence entries from that source until
within cap. Report in AUTO-REJECTED section.
Note: this cap operates at source level. The publisher
concentration check in Phase 1 operates at publisher level.
Both apply independently.

STEP 3 — Recommended removals (mandatory)
